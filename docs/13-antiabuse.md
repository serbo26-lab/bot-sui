# Antiabuse

Antiabuse — одна из самых важных систем Bot S-UI.

Она отслеживает превышение лимита устройств/IP и помогает администратору понять, используется ли подписка несколькими людьми.

Важно: antiabuse не видит “физические устройства” напрямую. Он анализирует IP-адреса и события подключения.

## Что считает antiabuse

Бот считает активные логические источники IP по подписке.

Пример:

```text
Лимит устройств/IP: 4
Активных источников: 5
→ подозрение на превышение
```

## Источники данных

Antiabuse собирает данные из двух мест:

```text
1. MAIN / S-UI server
2. Remote nodes
```

## MAIN antiabuse

На main-сервере бот читает журнал S-UI/sing-box.

Он ищет события подключения и сопоставляет IP, client name, protocol и timestamp.

## Remote node antiabuse

На remote node может работать collector.

Collector на node:

1. читает локальные sing-box логи;
2. собирает активные IP;
3. пишет summary-файл;
4. main-сервер периодически забирает summary.

Main потом объединяет node events с MAIN events.

## Периодичность

Типовая логика:

```text
MAIN scan: примерно раз в минуту
node collector: примерно раз в 2 минуты
main poll remote summaries: например раз в 5 минут
active IP TTL: например 10 минут
```

Это значит, что MAIN-данные обычно свежие быстрее, а remote node-данные могут отставать до следующего poll.

## Почему remote poll и TTL важны

Если active TTL = 10 минут, а main poll remote summaries = 15 минут, короткое превышение можно пропустить.

Компромисс:

```text
remote poll: 5 минут
active TTL: 10 минут
```

Так данные достаточно свежие, но старые IP не держатся слишком долго.

## Лимит устройств/IP

Источник лимита:

1. override в antiabuse-кейсе;
2. лимит тестового доступа;
3. лимит тарифа;
4. fallback из config.

## Dynamic IP dedupe

Провайдеры часто меняют IP, особенно мобильные.

Например:

```text
85.174.181.36
85.174.183.98
```

Это может быть одно устройство и один провайдер с динамической сменой IP.

Чтобы не отправлять ложные предупреждения, бот может группировать такие IP:

```text
85.174.*.* = один логический источник
```

Полные IP при этом сохраняются в evidence.

## Когда создается кейс

Кейс создается, когда активных логических источников больше лимита.

Пример:

```text
Устройств/IP сейчас: 5/4
```

Если после dynamic dedupe стало:

```text
логических источников: 4/4
```

warning не должен отправляться.

## Что показывает кейс

Кейс может показывать:

- client name;
- источник;
- статус;
- лимит IP;
- тип подписки;
- сервер;
- тариф;
- TGID;
- username;
- устройств/IP сейчас;
- IP за 1 час;
- IP за 24 часа;
- уникальных за retention;
- последнюю активность;
- risk score;
- свежесть MAIN/node данных;
- auto-disable status;
- warnings counters;
- ограничения;
- причины;
- источники/IP.

## Предупреждения

Если превышение активно, бот отправляет предупреждение админу и пользователю, если включены user warnings и есть TGID.

Предупреждения могут повторяться, например раз в 30 минут, пока превышение сохраняется.

## Счетчики предупреждений

В кейсе полезно показывать:

```text
за 24 ч
за 7 дней
за 30 дней
всего
последнее предупреждение
когда возможен следующий повтор
```

## Уведомление пользователю

Пользовательский текст должен быть без IP и технических деталей.

Пример:

```text
⚠️ По вашей подписке превышено количество разрешенных одновременных подключений.

Проверьте, не используется ли подписка на лишних устройствах или другими людьми.
```

## Ручное отключение

В кейсе есть ручные действия:

```text
⛔ Отключить на 10 мин
⏱ Свой срок
✅ Включить сейчас
```

Отключение должно применяться ко всей подписке: main/S-UI, MultiHop и remote node после sync/deploy.

## Auto-disable

Auto-disable — автоматическое временное отключение подписки.

Он должен быть выключен по умолчанию.

Безопасная логика:

```text
1. Обнаружено превышение.
2. Отправлено предупреждение.
3. Проверка повторяется несколько раз.
4. Перед отключением выполняется forced poll remote summaries.
5. Если данные node устарели — auto-disable не применяется.
6. Если превышение подтверждено — подписка отключается.
7. После auto-enable действует cooldown.
```

## Настройки auto-disable

Админ может настроить включение/выключение, длительность отключения, количество подтверждений и cooldown после auto-enable.

По умолчанию:

```text
auto-disable выключен
duration 10 минут
confirmations 2
cooldown 10 минут
```

## Evidence

Evidence — доказательная база.

Она нужна, чтобы админ мог увидеть время проверки, лимит, количество IP, grouped logical sources, raw IP, источники main/node, freshness, confirmation count и причину auto-disable.

Evidence лучше показывать отдельной кнопкой и хранить только последние записи, чтобы не раздувать кейс.

## Stale node data

Если node data старше допустимого порога, auto-disable не должен применяться.

В кейсе можно показывать:

```text
Данные node устарели — auto-disable не применяется до обновления.
```

## Что проверить

```text
1. MAIN IP попадают в кейс.
2. Remote node IP попадают в кейс.
3. Один IP на разных node считается один раз.
4. Dynamic IP dedupe работает.
5. Trial использует trial limit.
6. Paid использует tariff limit.
7. Manual/offline без TGID предупреждает только админа.
8. User warning приходит пользователю с TGID.
9. Ручное отключение работает.
10. Auto-disable выключен по умолчанию.
11. Evidence открывается.
```
