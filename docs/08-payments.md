# Платежи

Платежный модуль отвечает за покупку и продление подписок.

В текущей архитектуре Bot S-UI рассчитан на два основных способа оплаты:

```text
1. Telegram Stars
2. Platega / карта / СБП
```

Если нужно подключить другой платежный провайдер, потребуется отдельная адаптация кода.

## Где находится

Пользователь:

```text
🛒 Купить
🔄 Продлить
💳 Мои платежи
```

Админ:

```text
🛠 Админ → 💳 Платежи
```

## Telegram Stars

Telegram Stars — встроенная оплата Telegram.

Особенности:

- пользователь платит Stars внутри Telegram;
- цена Stars задается в тарифе;
- Stars можно включать/выключать по тарифу;
- если цена Stars 0 или Stars выключены, кнопка не показывается.

## Platega / карта / СБП

Platega используется для оплаты картой/СБП.

Обычно сценарий такой:

1. Бот создает payment record.
2. Бот создает платеж у провайдера.
3. Пользователь переходит по ссылке оплаты.
4. Провайдер подтверждает оплату.
5. Бот применяет подписку.
6. Пользователь получает ссылку.

## Покупка подписки

```text
пользователь → тариф → срок → способ оплаты → payment → success → create subscription
```

После успешной оплаты бот:

- создает клиента в S-UI;
- применяет тариф;
- задает срок;
- задает traffic/inbounds;
- создает remote credentials;
- ставит remote sync;
- добавляет external links;
- отправляет subscription link.

## Продление подписки

```text
пользователь → выбирает подписку → срок → способ оплаты → payment → success → renew
```

После оплаты бот должен продлить существующего клиента, а не создать дубль.

## Статусы платежа

```text
pending      — платеж создан, ожидает оплаты
processing   — платеж подтвержден, бот применяет действие
success      — подписка создана/продлена
failed       — ошибка или отмена
expired      — платеж устарел
```

## application_status

Отдельное состояние применения:

```text
not_applied
applied
apply_failed
reprocessing
```

Это нужно, потому что оплата и применение подписки — разные этапы.

## apply_failed

`apply_failed` означает:

```text
деньги прошли, но бот не смог создать/продлить подписку
```

Возможные причины:

- S-UI API недоступен;
- ошибка editbulk;
- ошибка remote sync;
- network timeout;
- неверный payload;
- неожиданная ошибка в коде.

## Retry

Retry повторяет применение платежа.

Retry должен быть идемпотентным: повторный retry не должен создавать дубль подписки.

## Approve вручную

Админ может вручную подтвердить платеж, если такая функция включена.

Это полезно, когда callback не пришел, провайдер подтвердил оплату отдельно или нужно выдать подписку вручную по payment record.

## result_payload

После успешной обработки бот может сохранить результат:

- client name;
- tariff;
- period;
- expiry;
- subscription link;
- Clash link;
- provider charge id;
- другие технические данные.

Если пользователь потерял сообщение, бот может показать результат повторно.

## Мои платежи

Пользовательский раздел показывает историю платежей: дату, сумму, способ оплаты, статус и тип операции.

## Админские фильтры платежей

Админка может иметь фильтры:

```text
все
успешные
ожидают
ошибки
Platega
Stars
Platega ожидают
Apply failed
```

## Что нельзя ломать

Платежи — критичный модуль.

Не менять без отдельной проверки:

- payload;
- result_payload;
- apply_failed;
- retry;
- duplicate protection;
- provider status mapping;
- Stars invoice handling;
- S-UI create/renew payload.

## Чеклист проверки платежей

```text
1. Покупка 30 дней картой/СБП.
2. Покупка 365 дней картой/СБП.
3. Покупка Stars.
4. Продление картой/СБП.
5. Продление Stars.
6. Реферальная скидка.
7. Pending payment.
8. Failed payment.
9. apply_failed → retry.
10. Повторный callback не создает дубль.
11. Пользователь видит ссылку.
12. Админ видит платеж.
```
